## Custom Project

### Dataset

The Kafka producer `kafka_producer_case.py` uses the unmodified `sales.csv` dataset.

This dataset contains simulated sales transaction records.

Each record represents a single customer purchase.

Fields in the dataset include:

- `order_id`: unique identifier for the transaction
- `datetime`: timestamp for the purchase event
- `region_id`: geographic region
- `currency_code`: currency used
- `product_id`: purchased product ID
- `unit_price`: price per unit
- `quantity`: number of units purchased
- `is_online`: (T/F) whether the purchase was online
- `customer_id`: unique customer identifier
- `is_new_customer`: whether the customer is new
- `device_type`: device used
- `payment_method`: payment type
- `referral_source`: traffic source
- `discount_code`: optional discount code
- `customer_note`: optional customer comment

The dataset itself was not altered.
All enhancements were applied downstream in the consumer.

### Kafka Messages

The producer `kafka_producer_case.py` sends raw sales records from the dataset to Kafka.

The Kafka topic for this project is streaming-06-scenarios-sailors.

### Consumer Processing

The consumer receives each Kafka message as a dictionary
and processes it through a structured pipeline.

For each message, the consumer:

- Reads messages from Kafka
- Consumes messages until a timeout or maximum message limit is reached
- Logs the raw message and validation status
- Validates required fields against a data contract
- Enriches valid records using reference data
- Computes derived fields
- Filters data based on reference data rules
- Writes accepted records to a CSV file and DuckDB
- Tracks and logs running statistics

Rejected records are logged and skipped.

### Results

When running the producer and consumer:

- All 10 valid messages were consumed successfully
- Invalid or filtered records were skipped and logged
- Enriched records were immediately visible in both CSV and DuckDB outputs
- Derived fields correctly reflected reference data logic
- Running statistics updated in real time as messages were processed

The DuckDB output enabled direct querying of enriched streaming results.

### Interpretation

The Kafka pipeline showed how data flows continuously from producer to consumer.
Each message is validated, enriched, stored, and visualized in real time.

Observing Kafka messages made the data pipeline easier to understand, which shows how
raw sales events become structured records through each stage.

From a business perspective, this stream could support real-time monitoring of sales performance.

It could

- help identify trends, spikes, or drops as they happen
- support faster decisions around pricing, promotions, and regional performance
